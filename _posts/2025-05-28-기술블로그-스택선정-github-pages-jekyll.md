기술 블로그를 개설하면서 고민했던 스택 선정 과정을 정리했다. GitHub Pages + Jekyll + MMistakes 조합을 선택하게 된 이유와 경험을 공유한다.

## 스택 선정 과정

### 내 요구사항
- **비용**: 무료 또는 최소한의 비용
- **시간**: 블로그 설정/꾸미기에 시간 최소화  
- **코드 표현**: 다양한 기술 스택의 코드 하이라이팅 지원
- **익숙함**: 개발자에게 친숙한 워크플로우

### GitHub Pages를 선택한 단순한 이유

기술 블로그를 시작하면서 가장 중요하게 생각한 것은 **블로그 설정에 시간을 많이 쓰지 않는 것**이었다. 글 쓰는 것도 어색한데, 블로그 꾸미기에 힘을 다 빼면 정작 글을 쓰기도 전에 지칠 것 같았다.

여러 옵션을 검토했지만:
- **워드프레스**: 호스팅 비용 + 초기 광고 수익으론 적자 확실
- **Velog/Medium**: 광고 수익 기대 어려움
- **GitHub Pages**: 완전 무료 + 개발자에게 가장 익숙한 환경

결국 가장 익숙하고 비용 부담 없는 **GitHub Pages**를 선택했다.

### 정적 사이트 생성기 선택

GitHub Pages에서는 직접 HTML/CSS를 작성하거나 정적 사이트 생성기를 사용할 수 있다. 당연히 **정적 사이트 생성기**를 선택했다. 직접 코딩하면 또 꾸미기에 시간을 쓰게 될 테니까.

다행히 정적 사이트 생성기도 내 요구사항을 모두 충족했다:
- ✅ **무료**: 오픈소스 도구들
- ✅ **시간 절약**: 검증된 테마 활용 가능
- ✅ **코드 표현**: 마크다운 + 코드 하이라이팅 기본 지원
- ✅ **익숙함**: Git 기반 워크플로우

### Jekyll을 선택한 이유

GitHub Pages에서 **Jekyll**은 특별한 위치에 있다:

1. **GitHub 공식 지원**: 별도 빌드 설정 없이 자동 배포
2. **설정 최소화**: `_config.yml` 하나면 기본 설정 완료
3. **테마 생태계**: 검증된 테마들이 많음
4. **마크다운 네이티브**: 기술 글쓰기에 최적화

다른 생성기들은 GitHub Actions 설정이나 별도 빌드 과정이 필요하지만, Jekyll은 푸시만 하면 끝이다.

### MMistakes 테마 선택

Jekyll 테마 중에서도 **Minimal Mistakes**를 선택한 이유:

1. **기술 블로그 특화**: 코드 하이라이팅, 목차, 카테고리 등 기본 제공
2. **문서화 완벽**: 설정 방법이 상세하게 정리되어 있음
3. **커뮤니티 활성**: 문제 해결 자료가 많음
4. **꾸미기 시간 최소화**: 기본 설정만으로도 깔끔한 디자인

결국 **"시간을 최소한으로 쓰면서 기술 블로그다운 모습"**을 만들 수 있는 조합이었다.

## 마무리

스택 선정 과정에서 가장 중요하게 생각한 것은 **비용보다는 시간**, **복잡함보다는 익숙함**이었다. GitHub Pages + Jekyll + MMistakes 조합은 이 모든 요구사항을 충족시켜줬다.
